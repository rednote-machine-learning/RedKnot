4. FP8 离线/在线算子启用 (本环境特有, 必读)
--------------------------------------------------------------------------------
问题: transformers 5.x 的 FP8 matmul 在首次 forward 时要从 HF Hub 动态下载
      kernel (deep-gemm / finegrained-fp8)。本机 HF 被设离线 + Hub 上没有匹配
      torch2.9-cu128 的 finegrained-fp8 build (只有通用 torch-cuda)，且其符号名
      (w8a8_*) 与 transformers 期望 (matmul/batched/grouped) 不一致 -> 直接跑会
      报 OfflineModeIsEnabled 或 "missing required symbols"。

解决: 仓库新增补丁 test/srt/redknot/utils/fp8_offline_patch.py
      - benchmark 脚本在 import 前自动挂载 (FP8 模型路径自动启用,
        由 REDKNOT_FP8_TORCH_FALLBACK 控制, 默认对 *fp8* 路径开启)。
      - 优先使用本地缓存的 Triton kernel 并做 API shim (matmul/batched/grouped),
        即真正的融合算子路径; 失败则退回纯 PyTorch block 反量化 (正确但慢)。
      - 修复了上游 fp8_act_quant 对全零激活块的除零 -> NaN bug
        (这是 FP8 输出退化成 '!!!!' / F1=0 的根因)。

依赖的本地 Triton kernel 缓存 (必须存在, 9 个纯 .py 文件, 无 .so):
  /root/.cache/huggingface/hub/models--kernels-community--finegrained-fp8/
      snapshots/<sha>/build/torch-cuda/
        __init__.py  _ops.py  act_quant.py  batched.py  grouped.py
        matmul.py  metadata.json  utils.py  finegrained_fp8/__init__.py

  若该缓存丢失, 在能访问 hf-mirror.com 时用 curl 重新下载到上面布局即可
  (sha 见 refs/main; resolve URL:
   https://hf-mirror.com/kernels-community/finegrained-fp8/resolve/<sha>/build/torch-cuda/<file>)

5. 环境变量 (运行时)
--------------------------------------------------------------------------------
HF / 离线相关:
  HF_ENDPOINT=https://hf-mirror.com
  HF_HUB_OFFLINE=1
  TRANSFORMERS_OFFLINE=1

GPU:
  CUDA_VISIBLE_DEVICES=0,1,2,3,4,5,6,7

脚本读取的 REDKNOT_* 环境变量:
  REDKNOT_MODEL_PATH         模型目录 (默认 35B-A3B; 跑 397B 必须显式设 FP8 路径)
  REDKNOT_CONFIG_TAG         配置标签, 397B 用 "qwen3.5-397B-A17B"
  REDKNOT_LONGBENCH_DIR      LongBench data 目录
  REDKNOT_EXPERTS_IMPL       MoE 专家实现; 离线用 "eager" (纯 python 循环, 走 fp8_linear)
  REDKNOT_FP8_TORCH_FALLBACK FP8 补丁开关 (默认对 fp8 路径自动=1)
  REDKNOT_N_SAMPLES          每个 config 的样本数 (默认 2)
  REDKNOT_MAX_NEW            每样本生成 token 数 (默认 48)
  REDKNOT_CONFIGS            "标签:数据集:n_chunk:chunk" (如 triviaqa@32K:triviaqa:4:8000)
  REDKNOT_DEVICE_MAP         device_map 策略 (默认 auto)
  REDKNOT_MAX_MEM_PER_GPU    每卡显存上限
  REDKNOT_MAX_CPU_MEM        CPU 内存上限 (防 memcg OOM)
  其余可调超参 (一般用 JSON 默认值即可):
    REDKNOT_DENSE_FULL_LAYERS  REDKNOT_FRAC_GLOBAL  REDKNOT_FULL_WINDOW
    REDKNOT_DENSE_PREFIX_LAYERS REDKNOT_DECAY_QUANTILE REDKNOT_SAFETY
    REDKNOT_MIN_WINDOW  REDKNOT_LINEAR_SEG
    REDKNOT_DEEP_MOE_START  REDKNOT_MOE_MASS_THRESH
    REDKNOT_OFFLINE_KV  REDKNOT_GLOBAL_OFFSET  REDKNOT_SEED

7. 已知注意事项 / 坑
--------------------------------------------------------------------------------
- .venv_tf5 的 import 很慢 (transformers 5.x 冷启动约 2-3 分钟), 建议后台运行 + 轮询日志。
- 权重加载 ~4-7 分钟 (FP8 379G, 8 卡分发)。
- eager MoE 在 32K 上下文下单样本前向很慢 (分钟级); 想快速验证可用 16K + 小 N。
- CPU 内存: 加载时峰值高 + page cache 会算进 memcg, 之前触发过 OOM-kill。
  若 OOM: 先清 page cache (sync; echo 1 > /proc/sys/vm/drop_caches) 再重跑。
- 启动建议用 setsid, 防止 shell 退出/超时杀掉长任务。
- 日志默认写到: test/srt/redknot/rag_logs/
- 验证产物: 每样本打印 "Baseline output" / "RedKnot output";
  结尾打印 Compute saving / TTFT speedup / 每数据集 F1 (PLOTROW 行)。
  正确运行时 baseline 与 RedKnot 输出应为正常文本 (出现 '!!!!' 即 FP8 NaN, 检查补丁)。
