# QPS Comparison Demo

## Figure Purpose

This figure compares average QPS per GPU under long-context inference for four model scales:

- Mistral-7B-Instruct (TP=1)
- Qwen3-32B (TP=2)
- Llama-3.3-70B-Instruct (TP=4)
- Qwen3.5-397B-A17B (TP=8)

The goal is to show how RedKnot improves serving throughput compared with full recomputation and lossy KV reuse baselines under different model architectures, context lengths, and tensor-parallel placements.

## Compared Methods

- **Recompute**: standard full prefill recomputation for every request. This is the throughput baseline.
- **CacheBlend (r=15%)**: offline KV reuse with top-15% token full recomputation. This is a lossy approximate reuse baseline.
- **ProphetKV (r=20%)**: offline KV reuse with top-20% token full recomputation. This is also a lossy approximate reuse baseline.
- **RedKnot (lossless)**: RedKnot uses model-aware attention/head reuse, carry-prefix recomputation, and sparse FFN execution while preserving exact/lossless behavior in this setup.

## Experimental Setup

All numbers are reported as average QPS per GPU. Different models are placed with different tensor parallelism degrees because their memory footprint and compute requirements differ:

| Model | Context Lengths | TP | Precision / Type |
|---|---:|---:|---|
| Mistral-7B-Instruct | 8K, 20K, 31K | 1 | Dense BF16 |
| Qwen3-32B | 8K, 16K, 32K | 2 | Dense BF16 |
| Llama-3.3-70B-Instruct | 16K, 64K, 128K | 4 | Dense BF16 |
| Qwen3.5-397B-A17B | 16K, 32K, 64K | 8 | Hybrid MoE FP8 |

The QPS model includes both prefill and decode cost. The script uses:

- Batch size: 32
- Decode length: 256 tokens
- Query tokens: 200
- Hardware assumption: H800-class GPUs
- Output metric: average QPS per GPU

## RedKnot Configuration

### Mistral-7B-Instruct

Mistral uses a chunk-based RedKnot design:

- Maximum context: 32K
- Chunk size: 4K
- Carry prefix: 2K
- Attention heads are treated as local-window heads
- Offline KV is generated per chunk and reused online
- Except for the first chunk, only the 2K carry prefix is recomputed
- FFN average sparsity: 60%

This setup makes RedKnot increasingly effective as context length grows because more chunks can reuse offline KV while avoiding full-context recomputation.

### Qwen3-32B

Qwen3-32B uses head-class based RedKnot:

- 64 layers total
- First 48 layers: all local-window attention heads
- Last 16 layers: mixed local, retrieval, and global heads
- Local window size: 4096
- Retrieval top-p: 0.9
- Sparse FFN skip fraction: 25% for deeper layers
- Tensor parallelism: TP=2

The main saving comes from replacing full attention in local heads with windowed computation and skipping selected FFN work in deeper layers.

### Llama-3.3-70B-Instruct

Llama-70B also uses head-class based RedKnot:

- 80 layers total
- First 60 layers: all local-window attention heads
- Last 20 layers: mixed local, retrieval, and global heads
- Local window size: 8192
- Retrieval top-p: 0.9
- Sparse FFN skip fraction: 25% for deeper layers
- Tensor parallelism: TP=4

The RedKnot gain becomes larger at 64K and 128K because attention recomputation cost grows quickly with context length.

### Qwen3.5-397B-A17B

Qwen3.5-397B-A17B is modeled as a hybrid MoE system:

- 60 layers total
- 15 full-attention layers
- 45 linear-attention layers
- Dense layers preserved for early layers
- Local attention fraction: 60%
- Local window size: 2048
- Linear-attention local window: 4096
- Sparse FFN skip fraction: 53% for deeper layers
- Tensor parallelism: TP=8

For this model, RedKnot combines attention savings with sparse MoE/FFN savings.

## Data Generation Process

The plotted values are generated by `plot_qps_4models.py` using analytical cost modeling plus measured/estimated serving parameters.

The process is:

1. Compute full recomputation FLOPs for each model and context length.
2. Compute CacheBlend and ProphetKV speedups using the offline KV reuse plus top-r% recomputation model.
3. Compute RedKnot speedup from model-specific attention/head and FFN savings.
4. Convert prefill FLOPs to prefill latency using H800 BF16 peak throughput and model-specific effective utilization.
5. Estimate decode latency from model weight bandwidth and KV-cache bandwidth.
6. Report QPS per GPU as the inverse of the dominant prefill/decode stage, normalized by TP.
7. Save the figure as both PNG and PDF.

For CacheBlend and ProphetKV, the speedup formula is:

```text
speedup = 1 / ((r + query_tokens / context_length + scoring_overhead) / recompute_efficiency)
```

where:

- CacheBlend uses `r = 0.15`
- ProphetKV uses `r = 0.20`
- `query_tokens = 200`
- `scoring_overhead = 0.05`
- `recompute_efficiency = 0.60`

## Data Analysis

RedKnot improves QPS over full recomputation for all four models and all tested context lengths. The improvement range is approximately:

- **Mistral-7B-Instruct**: 2.56x to 3.51x
- **Qwen3-32B**: 1.37x to 1.72x
- **Llama-3.3-70B-Instruct**: 1.31x to 2.20x
- **Qwen3.5-397B-A17B**: 1.75x to 1.84x

Overall, RedKnot achieves about **1.31x to 3.51x QPS improvement** over full recomputation across these settings.

The largest RedKnot gain appears on Mistral-7B at long context because the chunk-based carry-prefix design avoids most repeated prefill computation. The gain also increases with context length for Qwen3-32B and Llama-70B because local-window and retrieval-head savings become more important as full attention cost grows.

CacheBlend and ProphetKV can produce higher raw QPS in some dense-model settings because they aggressively reuse offline KV and only recompute a small selected token ratio. However, these methods are lossy approximations, while RedKnot is presented as the lossless method in this comparison. Therefore, the key comparison is that RedKnot consistently improves over full recomputation without relying on lossy token dropping or approximate KV reuse.
